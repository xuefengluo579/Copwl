## 一、项目启动（2026-06-30）

### 初始需求
- 开发老年陪伴AI的音频感知子系统
- 基于 w2v-SELD 实现声音事件检测（SED）
- 不做微调，先搭建推理流水线

### 关键决策

**决策1：模块化拆分**
- `音频识别/` — w2v-SELD 推理引擎（独立 Python 包）
- `音频接收转换/` — 音频采集与预处理管道（独立 Python 包）
- 两个模块解耦，采集和推理可部署在不同设备上

**决策2：双策略模型加载**
- fairseq API → torch.load state_dict fallback
- 原因：fairseq 和 PyTorch 版本强耦合，线上环境不可控

**决策3：Mock 模型测试**
- CI 环境无 GPU/无权重也能验证全流水线
- 91 个测试用例，覆盖预处理、后处理、异常处理

**决策4：配置三层体系**
- dataclass 默认值 → YAML 文件 → 环境变量覆盖
- 原因：配置项 ~10 个，不需要引入 Hydra 等重型框架

### 交付
- 33 个源文件，4173 行 Python
- 7 类事件检测（speech/fall/glass_break/knock/object_drop/footstep/door_slam）
- FastAPI REST 服务

## 二、音频采集管道（2026-06-30）

### 为什么独立

AudioPipeline 是 w2v-SELD 的上游，但两个模块职责不同：
- AudioPipeline：跨平台麦克风采集、重采样、AGC、通道转换
- SELD：深度学习推理

独立后可以各自单独部署、单独测试、单独迭代。

### 关键技术

| 技术点 | 选择 | 原因 |
|--------|------|------|
| 麦克风库 | PyAudio | 跨平台，社区成熟 |
| 重采样 | scipy.signal.resample | 抗混叠，比线性插值好 |
| AGC | 滑窗 RMS + 增益限幅 | 防止振荡，0.1~10x |
| 通道转换 | 伪 Ambisonics 合成 | 单麦→四通道的 HRTF 近似 |

### 伪空间合成原理

```
单声道 → 四通道 B-format:
  W  = 全向（原始信号）
  X  = 高通滤波 → 前方感知（高频直达声）
  Y  = 延迟 0.5ms → 左右时间差（ITD）
  Z  = 低通滤波 → 上下感知（地面反射）
```

## 三、微调训练方案（2026-06-30）

### 为什么不自己写训练框架

官方 `seld_wav2vec2` 仓库已有完整训练代码，重复造轮子无意义。我们只需要补数据准备工具。

### 数据方案

```
数据三来源:
  DCASE 公开数据集 → 基准能力
  自录场景音频     → 场景适配
  RIR 卷积合成     → 数据增强（解决四通道标注稀缺）
```

### 训练策略

- 分层学习率：backbone 1e-6 / task head 1e-4
- 防止灾难性遗忘：冻结预训练权重，只调任务头
- 按文件 70/15/15 切分，防止标签泄漏

## 四、核心难点讨论（2026-06-30 → 07-01）

### 实时音频采集难点
- 操作系统延迟抖动（非实时系统，10~30ms 不确定性）
- 重采样混叠（高频折叠到低频）
- AGC 反馈振荡（gain 过大→变小→过大循环）

### SELD 难点
- fairseq 依赖地狱（版本强耦合）
- Ambisonics B 格式通道顺序陷阱（ACN vs FuMa）
- SED 阈值艺术（按类别独立调参，跌倒下探到 0.3）
- GPU 预热开销（首帧 500ms+）

### 跨模块衔接问题
- 重复标准化冲突：Pipeline 和 Engine 都做标准化
- 帧长匹配：Pipeline 100ms ↔ Engine 100ms 刚好对齐
- 延迟预算：GPU 200ms（满足 <500ms）/ CPU 600ms（需优化）

## 五、深度学习三阶段（2026-07-01）

### 从识别到理解

```
第1层: 感知 (发现有什么)            ✅ 已完成
  AudioPipeline → SELD → events

第2层: 情境 (这发生在什么背景下)     方案已定，代码未写
  events + time + place + persona → context

第3层: 意图 (这会导致什么/该做什么)   方案已定，代码未写
  context + memory → action
```

### 模型更新策略

**核心原则**：神经网络权重不在用户端修改。

- 出厂模型不变（厂商控制更新）
- 适应能力放在决策层（Persona 统计学习）
- 云端聚合匿名数据 → 优化下一代模型 → 推送更新

### 三层适应架构

| 层 | 做法 | 是否训练 | 风险 |
|----|------|:--:|:--:|
| 统计学习（Persona） | 每日数据分析，指数移动平均 | ❌ | 无 |
| 轻量推理（融合分类器） | 规则 + 加权评分 | ❌ | 无 |
| Head-only 微调 | 冻结 backbone，只调几千参数 | ⚠️ 低风险 | 需要用户反馈标注 |

## 六、多模态融合（2026-07-01）

### 全局架构

```
AudioPipeline → SELD + Emotion → AudioPerception ─┐
                                                    ├→ FusionLayer → Decision
摄像头 → 姿态 + 表情 → VisualPerception ────────────┘
```

### 核心价值

- 听觉和视觉互相补充：单模态听力 fall 信心 0.4 → 加视觉 lying 确认 → 0.9+
- 矛盾时自动降级：听到 fall + 看到站立 → 可能只是东西掉了
- 情绪双源交叉验证：语音情绪 + 面部表情

### 融合矩阵

```
                    视觉:正常    视觉:异常
听觉:无异常          ✅ 正常      ⚠️ 关注
听觉:检测到异常       ⚠️ 降级      🚨 紧急
```

### 面试话术

> 「我定义了 AudioPerception 标准输出接口，融合层通过时间戳对齐两个感知源。用 4 象限交叉验证矩阵——双信号吻合时叠加置信度，矛盾时自动降级。新增任何感知模态只需实现对应的 Perception 接口。」

## 七、记忆系统设计（2026-07-01）

### 人类学习的启示

人类的听觉理解 → 匹配记忆模式 + 当前情境 → 判断。当前模型每次推理独立，没有「这个人平时什么样」的知识。

### 三层记忆架构（方案阶段）

```
短期记忆（分钟~小时）: 事件缓冲 + 时序模式匹配
  → "连续 knock+speech+knock = 有客人"

基线记忆（天~周）: 行为 Persona + 异常检测
  → "每天 8 点厨房有活动声，凌晨 3 点不该有脚步声"

情景记忆（长期）: 关键事件库 + 相似检索
  → "上次类似情况后来摔倒了"
```

### 个性化自适应

- 不是微调模型权重，是调整决策参数
- 夜间敏感度 ×1.5，活跃时段恢复正常
- 情绪连续下降 3 天 → 心理健康预警

## 十一、EmotionEngine 实现（2026-07-01）

### 新增模块

```
音频识别/src/emotion/
├── __init__.py     # 暴露 EmotionEngine, EmotionConfig, EmotionType, get_emotion_engine
├── models.py       # EmotionType 枚举（8 类）+ 中英文映射 + distress_weight
├── config.py       # EmotionConfig dataclass + 环境变量覆盖
└── engine.py       # EmotionEngine 主类（pipeline/原生双策略，单例模式）
```

### 技术决策

- **emotion2vec**：魔搭生态，wav2vec 同源架构，GPU < 50ms
- **8 类情绪**：happy/sad/angry/neutral/fear/surprise/disgust/pain
- **distress 权重**：pain=0.95 → happy=-0.2，直接供融合层消费
- **多通道降混**：4ch → mean(dim=0)，静音快速返回 None
- **中英文兼容**：emotion2vec 输出中文标签时自动映射为英文 EmotionType
- **双策略加载**：pipeline API → 原生 Model API fallback

### 测试

18 个新测试，累计 100 测试全部通过。
